This program lets you play Nine Men’s Morris in a graphical window against the computer. Click an empty point to place a stone during the opening phase. After all stones are placed, click one of your stones, then click its destination to move it. If you form a mill, click an opponent stone to remove it. The display shows whose turn it is, the current phase, the last move, whether the engine is thinking, and a scrolling game record. Use Reset to start a new game.

The program is a desktop implementation of Nine Men’s Morris written in F# with an Avalonia GUI. It combines three main parts: game-state logic, board rendering/input handling, and computer move selection.

The game model represents the board as a 24-element array, one entry per legal point on the Morris board. Each point is either empty or occupied by White or Black. The state also tracks whose turn it is, the current phase of play, how many stones each side has placed, whether a piece is currently selected for movement, whether a mill has just been formed, the winner if the game is over, and a status message for the UI.

The rules engine is implemented in the `MorrisBoard` module. It defines the adjacency graph for all 24 points and the full list of mill triples. The logic supports all main phases of Nine Men’s Morris: placing, moving, removing after forming a mill, flying when a side is reduced to three stones, and end-of-game detection. A move is applied by producing a new immutable `GameState` rather than mutating the old one. The functions `clickPoint`, `legalActions`, and `applyAction` are the main interfaces used by the GUI and AI. `clickPoint` interprets user clicks according to the current phase, while `legalActions` and `applyAction` provide a cleaner action-based interface for search.

The UI is built around a custom `BoardControl`. It draws the three nested squares, connecting lines, and the 24 legal board points. Each point is mapped to a fixed screen coordinate derived from the current control bounds. Empty points, occupied points, selected points, and removable targets are drawn with different fills or borders. Mouse clicks are converted into the nearest legal board point by checking the distance from the click position to each board node. The main window contains the board, status text, last-move text, a thinking indicator, a reset button, and a scrolling move record.

The move record is maintained separately from the board state. It stores structured turn records rather than only raw text, so a placement or move that forms a mill can be paired with the subsequent removal. The display then renders these turn records in compact notation, with one or two half-turns per line.

The computer player uses a negamax search with alpha-beta pruning. It searches legal actions generated from the current position and evaluates resulting states with a heuristic scoring function. The evaluator uses different priorities in placement and movement phases. It considers material count, completed mills, potential mills, immediate one-move mill threats, placement threats, double threats, blocked opposing pieces, and mobility. Extra weighting is used to discourage moves that leave the opponent with an immediate mill and to favor urgent defensive actions when needed. The search also extends slightly through removal phases and uses iterative deepening, with different target depths for placing and moving. Repeated positions along the current search path are treated as drawish to reduce cycling.

To keep the interface responsive, the engine search runs in a background task. When it is the computer’s turn, the UI first updates to show that the engine is thinking, then calls the search asynchronously, and finally marshals the chosen action back to the UI thread for display and application. That separation prevents the window from freezing during search and ensures the human player’s move is shown before the engine reply appears.
